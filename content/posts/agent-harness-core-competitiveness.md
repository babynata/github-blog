我先说结论：**判断一个 Agent 的核心竞争力时，我更看重的不是它用了哪个模型，而是它能不能把模型、harness、工具、权限和人类监督组织成一个稳定可控的执行系统。**

之所以这么看，主要有三层原因：第一，Agent 不是“会聊天的大模型”，而是一个能自己推进任务的系统；第二，真正决定体验上限和稳定性的，往往不是模型单点，而是 harness 这层系统组织能力；第三，harness 也不是越厚越好，它必须跟着模型能力和任务场景一起动态调整。下面我按这三点展开。

## 先把 Agent 看清楚：它不是一个单纯的模型壳

我觉得讨论 Agent 产品之前，第一步不是急着谈模型，而是先把“Agent 到底是什么”讲清楚。

Anthropic 在 **2024 年 12 月 19 日** 发布的《Building effective agents》里，把 workflow 和 agent 明确区分开了。前者是代码预先写好路径，模型按既定流程走；后者是模型自己决定怎么推进任务、怎么调用工具、什么时候继续、什么时候停下来问人。OpenAI 在它的 agent 指南里也给了非常接近的定义：Agent 不是把 LLM 接进应用就算完成，而是让模型去管理工作流执行，并在工具和 guardrails 之内独立完成任务。

这件事看起来像概念澄清，其实直接决定了我们该怎么看 Agent。

如果一个产品只是一个更会回答问题的聊天界面，那它的优劣确实很大程度上取决于底层模型。但如果它是一个真正的 Agent，那么它面对的就不是“回答得像不像”，而是“任务能不能做完、过程稳不稳、出错时能不能回退、什么时候该把决定交还给人”。

说白了，聊天模型像一个坐在柜台后面回答问题的人；Agent 更像一个真的下场办事的人。前者重要的是表达和知识，后者重要的是规划、执行、观察、修正，以及知道什么时候该停下来确认。

所以如果我要判断一个 Agent 的核心竞争力，我不会先问“它背后是不是最强模型”，我会先问：**它到底有没有把模型变成一个可执行、可校验、可控风险的系统。**

## 真正拉开差距的，往往不是模型单点，而是 harness 这层组织能力

我之所以会把判断重心放到 harness，不是因为模型不重要，而是因为 Agent 一旦进入真实任务，光有模型能力是不够的。

Anthropic 在 **2026 年 4 月 9 日** 发布的《Trustworthy agents in practice》里，把一个 agent 拆成四层：`model`、`harness`、`tools` 和 `environment`。这个拆法我很认同，因为它一下子把“能力从哪里来”和“风险从哪里来”都讲清楚了。模型提供智能，harness 提供指令和 guardrails，工具决定它能碰到什么系统，环境决定它能接触到什么数据与权限。任何一层出问题，最后交付出来的都不会是一个好 Agent。

这里面最像产品竞争力、也最容易被低估的，就是 harness。

harness 不是一个抽象概念。落到工程上，它通常至少包括这些东西：

- 上下文怎么组织，哪些信息该进窗口，哪些该外置存储
- 任务怎么拆，什么时候允许模型自主规划，什么时候强制走固定步骤
- 工具怎么定义，返回什么信息，哪些操作能合并成一个更自然的动作
- 权限怎么控，哪些动作默认允许，哪些动作必须经过人确认
- 状态怎么持久化，执行中断后能不能恢复，失败后能不能重试
- 评测怎么做，出了问题以后能不能知道错在哪一层

我越来越觉得，很多 Agent 产品之间的差距，就藏在这些“看起来不性感”的地方。模型像大脑，但 harness 更像操作系统和中层管理。它决定这颗大脑在真实世界里到底是高效办事，还是一路踩坑。

Anthropic 在 **2025 年 9 月 11 日** 的《Writing effective tools for agents》中也讲得很直白：Agent 的有效性，很大程度上取决于你给它什么工具，以及这些工具是不是按 Agent 的理解方式被设计、测试和评估过。换句话说，真正的竞争力不是“有工具”三个字，而是工具边界清不清楚、返回的信息是不是高信号、评测是不是能持续逼近真实任务。

如果一定要打个生活里的比方，我会觉得模型像一个很聪明的厨师，而 harness 像后厨的组织系统。菜能不能做出来，当然跟厨师手艺有关；但备料顺不顺、单据准不准、火候有没有人控、出错能不能补救，决定了这家店最后能不能稳定出餐。用户最终买单的，往往是整套出餐能力，而不是厨师履历本身。

## 但 harness 也不是越厚越好，它必须跟着模型和场景一起变

不过说到这里，我也不想把结论写成另一个极端，好像只要把 harness 做厚，Agent 就自然有护城河了。我不这么看。

Anthropic 在 **2026 年 4 月 8 日** 的《Scaling Managed Agents》里提醒了一件很重要的事：harness 里编码的，往往是“模型现在还做不到什么”的假设；而这些假设会随着模型变强很快过时。这句话我特别认同，因为它意味着真正优秀的 harness，不是把越来越多规则永远焊死在系统里，而是用尽量稳定的接口，把那些需要随模型演进而调整的假设留在可更新的层。

这也解释了为什么我不会把“模型”和“harness”看成对立面。更准确地说，两者是互相约束的。

- 模型能力弱、幻觉多、工具使用不稳时，需要更强的兜底、更细的权限边界、更密的检查点
- 模型能力更强、任务理解更好时，可以把一部分死板流程从系统里拿掉，让它承担更多规划和判断
- 但无论模型多强，权限、环境隔离、失败恢复和人类介入点都不会自动消失

OpenAI 的 agent 指南和 Anthropic 的《Building effective agents》其实都在强调同一件事：不是所有任务都值得上 Agent。只有当任务足够开放、固定规则很难覆盖、工具反馈可用，而且错误成本有办法控制时，Agent 才真正划算。路径稳定的任务，workflow 往往更便宜、更可预测；路径不稳定、需要边做边判断的任务，才更适合让 Agent 接管更多执行权。

所以如果把问题收束回具体的 Agent 产品，我更关心的就不是一句“它是不是 Agent”，而是另外几个更具体的问题：

- 它把哪些任务交给模型自由决策，哪些任务仍然保留确定性流程
- 它的工具、权限和环境是不是围绕真实任务设计的，而不是为了看起来功能很多
- 它有没有把失败恢复、人工确认和持续评测做成系统能力
- 它的 harness 能不能随着模型进步而变轻，而不是随着历史包袱越来越重

在我看来，这些问题的答案，才真正决定了一个 Agent 产品能不能形成长期竞争力。

## 写在最后

所以回到最开始那个问题，当我判断一个 Agent 的核心竞争力时，我现在更愿意把结论说得克制一点，也说得准确一点：**它的竞争力当然包含模型能力，但真正能把模型能力兑现成稳定体验的，往往是 harness、工具、环境和人类监督共同组成的系统设计。**

也正因为这样，我不会用“模型强不强”去替代“产品强不强”，也不会用“做了很多确定性封装”去替代“系统真的可靠”。一个成熟的 Agent，不是把模型包得越厚越好，也不是把权限放得越开越先进，而是能在自主性、控制力、成本和风险之间，找到一个持续可优化的平衡点。

如果一个 Agent 真要建立自己的核心竞争力，我会觉得它最该证明的，不是“我接了哪个最强模型”，而是**我能不能把模型变成一套长期可迭代、可交付结果、也可被用户信任的执行系统。**
